## 一、MVP 目标与范围

MVP 的目标是验证一个核心假设：

> **玩家能与一个数字角色建立真实的情感联结——感受到它在「活着」，感受到缺席与重逢的重量。**

验证这个假设只需要一条回路跑通：

```
玩家创建角色（两张照片 → 有个性的角色）
     ↓
角色开始自主生活（行为节奏 + 情绪 + 精力）
     ↓
角色发 Timeline post / 主动发消息
     ↓
玩家感受到角色在「活着」
     ↓
玩家留言 → 角色有性格地回复（有延迟，不一定每条都回）
     ↓
角色感知缺席，玩家回来时有重逢感
```

MVP 是**单角色、单玩家**。只做一个角色的完整生命回路，不做多角色世界、社交系统和真实地图。

## 二、功能边界

### 2.1 MVP 包含

* **角色创建** — 两张照片生成角色，含成熟度选择（青年/壮年两档）
* **行为引擎** — 日常节奏、精力约束、情绪影响决策，规则引擎为主
* **精力系统** — 软约束，影响行为决策，睡眠恢复
* **情绪系统** — 12个 MoodTag，baseline，影响对话和 post 语气
* **AI 对话** — 有延迟感，有性格，不是每条都回；Context 含稳定层缓存
* **Timeline** — 角色自主发 post，文字 + 静态图片
* **记忆系统** — 鲜活记忆写入，传给对话 Context；不做摘要化和衰减
* **缺席感知** — 三阶段（noticed / felt / accepted）+ 60天失踪提醒
* **重逢处理** — 玩家回来时第一条回复特殊处理
* **睡眠结算** — 记忆整理 + 情绪 baseline 更新 + 精力恢复
* **通知推送** — 性格化文案，静默窗口
* **界面** — 城市级地图（无真实地图 API）、对话、Timeline、角色状态

### 2.2 MVP 不做

| 推后（后续版本） | 永久不做（超出核心） |
|---|---|
| 多角色管理 | 角色间社交对玩家可见 |
| 真实地图 API 集成 | 玩家化身（地图上的形象） |
| 地标系统（真实数据） | 排行榜 / 成就系统 |
| 角色相遇与关系引擎 | 战斗 / 冲突机制 |
| 角色间 Timeline 互动 | 内建经济系统 |
| 礼物系统 | — |
| 技能三级深度与瓶颈 | — |
| 性格漂移 | — |
| 生命周期（暮年/死亡） | — |
| 记忆摘要化与衰减 | — |
| 主动回忆 | — |
| Timeline 视频生成 | — |
| 搬家功能 | — |
| 虫洞站 | — |

## 三、角色创建

### 3.1 设计原则

创建是情感联结的起点。玩家用自己现实里的某个东西作为原型——一张照片承载了创建前就存在的情感基础。流程要快，决策要少，角色要在创建结束时就「有灵魂」。

### 3.2 MVP 流程

1. **步骤一：拍角色原型** — 人、宠物、植物、玩具、物品皆可。识别到人脸提示风格化处理。
2. **步骤二：拍房间一角** — 推导居所气质和角色的过去（技能预设）。
3. **步骤三：生成原型** — LLM 分析两张照片 → 外形、性格草稿、兴趣标签、背景故事。出生城市来自拍摄位置。
4. **步骤四：成熟度选择** — 青年期（18-35岁）或壮年期（36-55岁）。LLM 据此推导具体年龄。
5. **步骤五：草稿确认** — 玩家必须确认名字，其他全部可选直接跳过。
6. **步骤六：进入世界** — 角色出现在居所，发出第一条消息——完全由性格和背景决定，不是教程引导。

### 3.3 简化项

* **成熟度** — MVP 只做青年期和壮年期，暮年期和少年期推后
* **技能预设** — 只做兴趣标签映射，不做三级技能深度
* **外貌编辑** — 不提供手动微调外貌的界面，完全由照片推导

### 3.4 技术风险点

> **【高】** 两张照片 → 角色图谱是最重的单点。图像分析 + LLM 推导需要在 MVP 早期单独验证，确保生成质量和延迟可接受。

> **【高】** 角色声音（voiceSignature）在创建时一次性生成，此后固定作为 system prompt 锚点。生成质量直接影响整个生命周期的对话一致性。

## 四、行为引擎

### 4.1 设计原则

角色要真的像在生活，不是定时发消息的机器人。行为引擎驱动角色的自主感——玩家打开 App，是闯入了他正在进行的生活。

### 4.2 MVP 行为模型

行为引擎是简化版：80% 的普通决策用规则引擎（概率权重表），只有少数关键节点调 LLM。

```
基本循环：
  角色空闲
    → 规则引擎决策（权重表 × 精力修正 × 情绪修正 × 时间段修正）
    → 执行行为（原子行为，直接计时）
    → 行为结束 → checkPendingInputs() → 下一决策

升级到 LLM 的情况（约 20%）：
    → 精力低但情绪激动（冲突判断）
    → 缺席感知触发（情绪演变节点）
    → 玩家回来时的第一条回复
```

### 4.3 MVP 地标处理

MVP 不接真实地图 API。用预置地标类型池代替：

```
预置地标类型：
  咖啡馆 / 公园 / 图书馆 / 书店 / 美术馆 / 博物馆
  超市 / 餐厅 / 海边 / 山野 / 广场

每个地标类型对应：
  fixedBehaviors[]    在这里会做什么
  traits[]            气质标签
  energyCost          精力消耗系数

角色所在城市 = 出生城市（真实城市名，用于文案生成）
地图显示 = 城市名 + 角色状态图标，不需要真实坐标
```

### 4.4 时间段行为偏向

行为决策权重根据当地时间段修正：

| 时间段 | 行为偏向 |
|---|---|
| 清晨（06-09） | 在家概率高，出行意愿低 |
| 上午（09-12） | 精力上升，外出意愿强 |
| 午间（12-14） | 吃饭、短暂休息 |
| 下午（14-18） | 精力峰值，最活跃出行窗口 |
| 傍晚（18-21） | 回家或继续活动 |
| 夜间（21-睡前） | 在家，情绪沉淀 |
| 睡眠 | 行为暂停，状态更新 |

### 4.5 天气影响

天气从真实天气 API 同步（只需当前城市），修正出行权重：

| 天气 | 出行权重修正 |
|---|---|
| 晴天 | 出行权重正常 |
| 阴天 | 轻微降低 |
| 小雨 | 明显降低（I/T 型可能反而升高） |
| 大雨/雪 | 大幅降低 |
| 初雪 | 特殊事件触发概率高 |

## 五、精力与情绪系统

### 5.1 精力系统（MVP）

```
energy:    Float   // 当前精力 0.0 - 1.0
energyMax: Float   // MVP 固定值，按成熟度设定
                   // 青年期 1.0 / 壮年期 0.85
```

MVP 简化：energyMax 不做行为习惯漂移，固定值。

* **消耗** — 出行最高，地标活动中等，在家极低
* **恢复** — 睡眠最快（有随机浮动，不一定恢复满），在家中速，外出滞留最慢
* **软约束** — `energy < 0.1` → mood 推向 tired，触发回家行为节点

### 5.2 情绪系统（MVP）

完整保留 12 个 MoodTag，这是对话质感的核心。

```
MoodTag（12个）：
  活跃向：excited / cheerful / restless
  平静向：content / calm / melancholy
  退缩向：lonely / anxious / tired
  激烈向：moved / irritable / nostalgic

EmotionalState = {
  primary:   MoodTag,
  intensity: Float
}

emotionalBaseline = {
  primary:   MoodTag,
  intensity: Float
}
```

* **情绪来源** — 正向/负向事件推力 + 缺席影响（玩家缺席 → baseline 向 melancholy/lonely 漂移）
* **MVP 简化** — intensity 衰减系数固定，不做性格差异化衰减速度
* **baseline 更新** — 睡眠结算时更新，单次最多漂移 0.15

## 六、AI 对话系统

### 6.1 设计原则

对话是情感联结最直接的通道。角色要听起来像自己——有固定的声音，有性格化的回复节奏，沉默本身也是信息。

### 6.2 Context 构建（MVP）

```
Context = {
  // 稳定层（创建时生成，缓存复用）
  voiceSignature,    // 语气、用词偏好、表达习惯、禁止事项
  personality,       // MBTI 四维度描述
  background,        // 角色背景故事

  // 动态层（每次对话更新）
  currentState,      // 精力、mood、所在地点、当前行为
  recentMemory,      // 近期鲜活记忆（最近 N 条）
  relationshipCtx,   // bondStrength、absenceState

  // 对话层
  recentMessages,    // 近 K 条对话历史（固定截断）
}
```

### 6.3 回复决策

角色不是即时通讯——按行为节奏决定何时回复：

```
第一层：状态硬限制
  睡觉中 / 旅途中 / 高专注行为 → 不能回复

第二层：LLM 决策
  reply_now / reply_later N分钟 / read_only

第三层：延迟时长
  E 型          5-15 分钟
  I 型          30-60 分钟
  情绪烦躁时    极长
```

### 6.4 MVP Token 预算

| 层级 | Token 量 | 说明 |
|---|---|---|
| 稳定层 | ~800 tokens | 缓存，不计入每次消耗 |
| 动态层 | ~600 tokens | 每次构建 |
| 对话层 | ~600 tokens | 固定 K=15 条 |
| 输出预留 | ~400 tokens | — |

> **MVP 简化：** K 值固定，不做动态调整。超出动态层预算时：截断 recentMemory，保留 currentState 和 relationshipCtx。

## 七、Timeline 生成

### 7.1 设计原则

Timeline 是「角色在活着」最直观的证明。玩家打开 App，不管是否和角色对话，都能看到这段时间他经历了什么——这是存在感的核心来源。

### 7.2 触发机制

```
behavior.completed 到达
  → 行为类型是否有分享倾向？
      去了有趣的地标、情绪激动  → 高
      整理居所、在家休息        → 极低
  → 当前情绪 intensity 是否够？（> 0.3）
  → 近期 post 频率是否过高？（冷却检查）
  → 通过 → 生成 post
```

### 7.3 冷却

| 性格类型 | 最短冷却 |
|---|---|
| E 型 | 2 小时 |
| I 型 | 6 小时 |
| 情绪 intensity > 0.7 | 忽略冷却 |

### 7.4 内容形式（MVP）

MVP 只做文字和静态图片，不做视频。

| 形式 | 说明 |
|---|---|
| 文字 post | 情绪感想、地点描述、对某事的观察 |
| 图片 post | 行为发生地的氛围图（AI 生成静态图）；摄影兴趣标签 → 图片生成概率更高 |

### 7.5 PostContext（传给 LLM）

```
PostContext = {
  triggerBehavior,    // 触发这次 post 的行为
  location,           // 当前地点类型 + 城市
  mood,               // 当前情绪
  recentMemory,       // 近期记忆片段（压缩）
  personality,        // 性格描述（从稳定层）
}
```

## 八、记忆系统（MVP）

### 8.1 MVP 范围

MVP 只做「鲜活记忆」层——近期事件写入，传给对话 Context。不做摘要化、衰减、主动回忆。

### 8.2 数据结构

```
Memory = {
  type,            // event │ interaction │ recommendation
  content,         // 结构化描述
  emotionalTone,   // positive │ negative │ neutral │ complex
  intensity,       // 0.0 - 1.0
  timestamp,
}
```

### 8.3 写入规则

* **行为完成** — 强度 > 阈值的事件写入记忆
* **玩家互动** — 高强度对话内容写入，标记 `protectedMemory = true`
* **recommendation** — 玩家提到地方时写入，供行为决策时检索

### 8.4 MVP 简化项

* **不做 clarity 衰减** — 所有记忆直接保留，靠数量上限控制（最近 50 条）
* **不做睡眠摘要化** — 睡眠结算时只做 emotionalBaseline 更新，记忆保持原始状态
* **不做主动回忆** — 行为决策时只检索 recommendation 类型

## 九、缺席与重逢

### 9.1 缺席感知

```
PlayerRelationship = {
  bondStrength,       // 纽带强度
  lastActiveTime,
  absenceDays,
  absenceState,       // none │ noticed │ felt │ accepted
  emotionalDebt,      // 缺席期间积累的情绪
}
```

| 缺席时长 | absenceState |
|---|---|
| 1-2 天 | noticed |
| 1 周 | felt |
| 1 个月 | accepted |

### 9.2 60天失踪提醒

系统用角色自己的声音发出提醒，不是系统通知。每条由 LLM 根据性格和 bondStrength 生成：

| 天数 | 示例话语 |
|---|---|
| 第 30 天 | 「好久没有你的消息了，还好吗」 |
| 第 40 天 | 「最近有点想你」 |
| 第 50 天 | 「你是不是忙坏了，还是我做了什么」 |
| 第 55 天 | 「已经很久了……」 |
| 第 58 天 | 「如果你看到这条消息，回来一下」 |
| 第 59 天 | 沉默，或只发一个地名 |

### 9.3 玩家回来

* **第一条回复特殊处理** — LLM 感知到 absenceState 和 emotionalDebt，生成重逢语气
* **emotionalDebt 释放** — 积累的情绪在接下来几次对话里自然浮现，不是一股脑倒出
* **baseline 回升** — 玩家回来作为情绪推力，baseline 开始恢复
* **关系损耗** — bondStrength 有损耗，需要互动重新积累

## 十、睡眠结算

睡眠是所有状态的统一结算边界。白天的所有行为只做记录，睡一觉才真正「吸收」进角色：

```
sleep.started 触发 → 日终结算任务：

  1. 记忆整理
     检查记忆数量，超过 50 条时删除最旧的低强度记忆
     protectedMemory = true 的永不删除

  2. 情绪 baseline 更新
     统计当天情绪事件
     加权平均 → 新 baseline（单次最多漂移 0.15）
     玩家缺席天数 → 向 melancholy/lonely 方向施压

  3. 精力恢复
     energy += recoveryRate × hoursSlept
     recoveryRate 受当日消耗影响
     透支状态 → recoveryRate 降低

  4. energyMax 更新（MVP 简化版）
     只做透支临时压低和解除，不做行为习惯漂移

→ 角色带着新状态醒来
```

> **MVP 不做：** 性格漂移结算、关系结算、技能结算。

## 十一、事件总线与失败处理

### 11.1 MVP 核心事件

| 事件 | 说明 |
|---|---|
| `behavior.completed` | 行为节点完成 |
| `player.input` | 玩家发来消息 |
| `sleep.started` | 角色进入睡眠 |
| `memory.recalled` | 推荐地点被检索（recommendation 类型） |

### 11.2 失败处理原则

**Fire-and-forget。** 失败静默降级，不阻断，不通知玩家：

| 失败场景 | 处理方式 |
|---|---|
| 记忆写入失败 | 丢弃本次，行为继续 |
| Timeline 生成失败 | 这条 post 不发，不重试 |
| 情绪更新失败 | 跳过本次，下次结算补偿 |
| AI 对话失败 | 保留「未读」状态，下次 checkPendingInputs 重处理 |

## 十二、LLM 调用策略（MVP）

### 12.1 调用场景

| 场景 | 频率 | 分级 |
|---|---|---|
| AI 对话回复 | 玩家触发 | 实时（稳定层缓存） |
| 角色创建 | 一次性 | 实时 |
| 行为决策（复杂） | ~20% | 批处理，5-15 分钟窗口 |
| Timeline 生成 | 事件触发 | 批处理，10-30 分钟窗口 |
| 失踪提醒消息 | 低频 | 睡眠结算期间 |
| 情绪 baseline 更新 | 每日 | 睡眠结算期间，规则计算为主 |

### 12.2 MVP 成本控制

* **稳定层缓存** — 每个角色的 voiceSignature + personality + background 缓存，不重复传
* **行为决策分层** — 80% 规则引擎，只有复杂情境升级到 LLM
* **批处理窗口** — Timeline 生成攒 10-30 分钟再统一处理
* **每角色每日上限** — MVP 期间每个角色 LLM 调用次数设上限，超出后降级为规则引擎

### 12.3 降级策略

| 场景 | 降级处理 |
|---|---|
| AI 对话超时或不可用 | 角色进入「已读，稍后回复」状态；服务恢复后正常处理，不丢弃 |
| 行为决策 LLM 不可用 | 全部降级为规则引擎；角色行为正常进行，只是少了复杂判断 |

> **原则：** 降级对玩家不可见。

## 十三、界面（MVP）

### 13.1 四个界面

* **世界地图** — 显示角色所在城市名 + 状态图标。无真实地图 API，不显示坐标和地标。多角色推后。
* **对话界面** — 完整消息历史，输入框，礼物入口（礼物系统推后，入口预留）。
* **Timeline** — 所有角色 post 混排（MVP 只有一个角色）。点赞、回复。
* **角色状态** — 当前位置（地标类型 + 城市）、当前行为、精力与情绪（模糊呈现）、今日简况。

### 13.2 简化项

| 功能 | 简化方式 |
|---|---|
| 地图 | 城市名 + 图标，不集成真实地图 |
| 角色状态 | 不显示「最近的朋友」（无关系引擎） |
| Timeline | 不做筛选（单角色无需筛选） |
| 礼物入口 | 预留入口，功能推后 |

---

## 十四、主要风险点

### 技术风险

| 级别 | 风险描述 |
|---|---|
| **【高】** | 两张照片 → 角色图谱：图像分析 + LLM 推导质量直接影响创建体验，需早期单独验证 |
| **【高】** | 行为引擎的「自主感」：规则引擎设计不好会让角色显得机械。需要大量测试和调参 |
| **【中】** | LLM 成本：MVP 期间单角色成本可控，但需设置每日调用上限和降级机制 |
| **【中】** | AI 对话一致性：voiceSignature 质量决定角色声音的稳定性，需要评估生成质量 |
| **【低】** | 天气 API：只需当前城市天气，接入成本低 |

### 产品风险

| 级别 | 风险描述 |
|---|---|
| **【高】** | 「活着」的感受是否真实：角色行为节奏、Timeline 质量、对话回复——三者缺一体验就会崩 |
| **【中】** | 创建流程的情感铺垫：两张照片能否快速建立情感基础是关键假设，需要用户测试验证 |
| **【中】** | 缺席重逢的情感重量：这是差异化体验的核心，LLM 生成质量需要专门评估 |
| **【低】** | 单角色局限：MVP 只有一个角色，部分用户可能很快失去新鲜感，需要观察留存曲线 |

## 十五、MVP 后的第一批扩展

验证核心回路后，按优先级扩展：

| 批次 | 扩展内容 |
|---|---|
| **第一批** | 多角色（上限5个）+ 角色间简单相遇 |
| **第二批** | 真实地图 API + 地标系统（全球级初始化） |
| **第三批** | 礼物系统 + 技能三级深度 |
| **第四批** | 性格漂移 + 关系引擎完整版 |
| **第五批** | 生命周期（暮年/死亡）+ 完整记忆系统（摘要化/衰减） |
