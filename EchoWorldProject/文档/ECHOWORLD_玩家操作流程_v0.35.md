## App 整体布局

ECHOWORLD 是一款手机 App（iOS / Android）。界面底部有三个固定 Tab，构成 App 的主导航结构。

### 底部导航：三个 Tab

| Tab | 说明 |
|---|---|
| **World** | 默认首页，世界地图视图 |
| **Discover** | 所有角色的 Timeline 汇集 |
| **Setting** | 玩家信息与设置 |

### World Tab — 世界地图

App 启动后的默认视图。全屏地图，所有角色的头像图标显示在各自当前位置上。

* 角色头像图标承载简单状态信息：睡眠时图标暗淡，有未读消息时显示角标，情绪激烈时有细微视觉变化。
* 点击角色头像图标 → 进入该角色的对话页。
* 对话页顶部显示角色头像和名字，点击头像 → 进入角色状态页。
* 角色状态页有「发消息」入口 → 返回对话页。
* 多角色时，地图自动缩放到能同时看到所有角色的级别（同城 / 跨城 / 跨国 / 跨洲）。
* 消息通知通过角色头像图标上的角标体现，不设独立消息列表页面。

### Discover Tab — Timeline

所有角色的 Timeline post 按时间混排，是角色生活的可见切片。

* Post 流默认全部角色混排，可按角色筛选，只查看某个角色的 post。
* 每条 post 显示角色头像、发布时间（玩家时区）、所在地点、内容（文字 / 图片）。
* 点击 post 上的角色头像 → 进入该角色的状态页。
* 玩家可对 post 点赞或回复，角色能感知到。

### Setting Tab — 设置

玩家信息展示与所有设置项的入口。

| 分组 | 功能项 |
|---|---|
| **玩家信息区** | 头像 / 姓名 / 性别（点击可进入 Profile 编辑） |
| **账户** | Profile（修改姓名、头像、性别）· Change Password · Change Email · Sign Out |
| **通知** | Notification（推送开关 / 静默窗口） |
| **偏好** | Languages |
| **支持** | ECHOWORLD Docs · Customer Service |
| **法律** | About Us · Privacy Policy |

### 页面导航关系

```
World（地图）
  └─ 点击角色头像
       └─ 对话页
            └─ 点击顶部角色头像
                 └─ 角色状态页
                      └─ 发消息按钮 → 对话页

Discover（Timeline）
  └─ 点击 post 角色头像 → 角色状态页

Setting
  └─ Profile → 编辑玩家信息
  └─ 各设置项 → 对应设置页
```

## 0 账户管理

本章涵盖玩家账户的注册、登录、退出及账户设置相关用例。

### UC-0.1 注册新账户

| 字段 | 说明 |
|---|---|
| **参与者** | 新用户（未注册） |
| **前置条件** | 用户已下载并打开 App，尚未拥有账户 |
| **后置条件** | 账户创建成功，邮箱已验证，用户进入角色创建流程 |

#### 主成功场景

1. 用户在启动界面选择「创建账户」。
2. 系统显示注册表单，用户输入邮箱地址。
3. 用户设置密码（8位以上，包含字母和数字）。
4. 用户提交注册。系统校验邮箱格式和密码强度。
5. 系统向该邮箱发送验证邮件。界面显示「请查收邮件，点击验证链接后继续」，并提供「重发验证邮件」按钮（60 秒冷却）。
6. 用户点击邮件中的验证链接，账户激活。App 自动跳转回，进入玩家身份设置。
7. 玩家身份设置（注册后一次性引导，后续可在 Settings → Profile 修改）：
   * 姓名（必填，角色认识的「你」的名字）
   * 头像（拍照 / 相册选取 / 跳过使用系统默认）
   * 性别（男 / 女 / 其他 / 不透露）
8. 完成后进入角色创建流程（UC-1.1）。

#### 扩展场景

| 条件 | 触发情形 | 处理方式 |
|---|---|---|
| **4a** | 邮箱格式不合法 | 即时提示错误，阻止提交 |
| **4b** | 密码不符合规则 | 即时提示规则要求，阻止提交 |
| **4c** | 邮箱已被注册 | 提示「该邮箱已有账户」，提供「去登录」入口 |
| **5a** | 验证邮件未收到 | 提示检查垃圾邮件；60 秒后可点击重发 |
| **6a** | 验证链接已过期（24小时） | 提示「链接已过期」，重新发送验证邮件 |
| **7a** | 玩家跳过姓名填写 | 阻止继续，姓名为必填项 |
| **7b** | 玩家跳过头像和性别 | 允许，头像使用系统默认，性别记录为「不透露」，后续可在 Profile 修改 |

### UC-0.2 登录

| 字段 | 说明 |
|---|---|
| **参与者** | 已注册且已验证邮箱的用户 |
| **前置条件** | 用户已有账户，当前未登录 |
| **后置条件** | 用户登录成功，系统颁发 OAuth access token 及 refresh token，进入 App 主界面 |

#### 主成功场景（方式 A：密码登录）

1. 用户在登录界面选择「密码登录」，输入邮箱和密码。
2. 用户点击「登录」。
3. 系统验证邮箱和密码，颁发 token，进入世界地图。

#### 主成功场景（方式 B：验证码登录）

1. 用户在登录界面选择「验证码登录」，输入邮箱。
2. 系统向该邮箱发送 6 位验证码，有效期 15 分钟。
3. 用户输入验证码，点击「登录」。
4. 系统验证通过，颁发 token，进入世界地图。

> **注：** 验证码 60 秒后可重发，同一邮箱同一时间只有一个有效验证码。

#### 扩展场景

| 条件 | 触发情形 | 处理方式 |
|---|---|---|
| **A2a** | 密码错误 | 提示重试；连续 5 次失败后临时锁定账户 30 分钟 |
| **A2b** | 邮箱未注册 | 提示「该邮箱未注册」，提供「去注册」入口 |
| **A2c** | 邮箱未验证 | 提示「请先验证邮箱」，提供重发验证邮件入口 |
| **A2d** | 忘记密码 | 提示「可以用验证码登录，登录后在设置里修改密码」，不在此界面拦截 |
| **B3a** | 验证码错误 | 提示重试，验证码有效期内不限次数 |
| **B3b** | 验证码过期 | 提示「验证码已过期，请重新发送」 |

> **注：** 系统自动维护 refresh token，access token 到期后静默刷新，用户无感知。会话默认长期保持，用户无需反复登录。

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录 |
| **后置条件** | 当前设备 token 失效，用户回到登录界面；账户数据和角色数据完整保留 |

### UC-0.4 修改密码（Change Password）

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录 |
| **后置条件** | 密码更新成功；当前设备会话保持，其他设备 token 失效 |

#### 主成功场景

1. 用户进入 Settings → 账户 → Change Password。
2. 用户选择验证方式：
   * ① 输入当前密码
   * ② 发送验证码到当前邮箱（适用于忘记当前密码）
3. 系统验证身份。
4. 用户输入新密码，确认新密码。
5. 用户提交。系统更新密码，提示「密码已更新，其他设备需要重新登录」。

#### 扩展场景

| 条件 | 触发情形 | 处理方式 |
|---|---|---|
| **2a** | 当前密码输入错误 | 提示重试，连续 5 次失败锁定 30 分钟 |
| **4a** | 新密码不符合规则 | 即时提示规则要求 |
| **4b** | 两次输入不一致 | 即时提示「两次输入不一致」 |

### UC-0.5 修改邮箱（Change Email）

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录 |
| **后置条件** | 账户邮箱更新为新邮箱；旧邮箱失效 |

#### 主成功场景

1. 用户进入 Settings → 账户 → Change Email。
2. 用户选择验证方式：
   * ① 输入当前密码
   * ② 发送验证码到当前邮箱
3. 系统验证身份通过。
4. 用户输入新邮箱地址。
5. 系统向新邮箱发送验证链接（有效期 24 小时）。界面提示「请查收新邮箱的验证邮件」。
6. 用户点击验证链接，邮箱更新完成。系统通知：「您的账户邮箱已更新。」

|---|---|---|
| **4a** | 新邮箱格式不合法 | 即时提示错误 |
| **4b** | 新邮箱已被其他账户使用 | 提示「该邮箱已绑定其他账户」 |
| **5a** | 验证链接未点击，24小时过期 | 变更取消，旧邮箱仍有效，可重新发起修改 |

> **注：** 验证完成前，旧邮箱仍为账户邮箱，登录和找回密码均使用旧邮箱。

## 一 第一次来到这个世界

玩家第一次打开 App，完成注册并创建第一个角色。角色从现实里的某个东西生长出来——创建之前就有情感基础了。这不是配置一个游戏角色，是一段关系的起点。

### UC-1.1 创建第一个角色

| 字段 | 说明 |
|---|---|
| **参与者** | 已完成注册和邮箱验证的新用户 |
| **前置条件** | 用户已登录，尚未创建任何角色 |
| **后置条件** | 角色创建完成，角色进入居所，向玩家发出第一条消息，玩家进入世界地图 |

#### 主成功场景

1. 系统显示空世界地图，以玩家当前真实位置为中心。地图上没有任何角色，传递「世界在等待」的感觉。界面提示引导玩家创建第一个角色。
2. 玩家拍摄或从相册选取第一张照片（角色原型：宠物、植物、玩具、物品皆可；暂不支持以人类照片创建角色）。识别到人脸时，系统提示「暂不支持以人物照片创建角色，请更换照片」，阻止继续。
3. 玩家拍摄或选取第二张照片（房间的一个角落），用于推导居所气质和角色的过去。
4. 系统分析两张照片，生成角色原型。界面呈现等待状态——不是进度条，是有仪式感的等待，角色正在从那些照片里生长出来。
5. 玩家选择角色的起点生命阶段：少年期（12–17岁）/ 青年期（18–35岁）/ 壮年期（36–55岁）/ 中老年（56–70岁）。系统据此推导具体年龄。
6. 系统呈现草稿确认界面：外形预览 · 名字 · 推导年龄 · 性别（由照片推导，可修改：男 / 女 / 中性 / 未知）· 一句话背景 · 性格滑块 · 兴趣标签 · 出生城市。玩家必须确认名字，年龄可在所选阶段范围内微调，其余字段可选跳过。
7. 玩家确认，角色正式创建。世界地图以出生城市为中心展开，角色出现在居所。
8. 角色向玩家发出第一条消息——完全由性格和背景决定，不是系统引导语。部分角色可能同时在 Timeline 发布第一条 post。

> **注：** 等待期间（步骤 4）界面设计应传递仪式感，不应使用通用加载动画。

#### 扩展场景

| 条件 | 触发情形 | 处理方式 |
|---|---|---|
| **2a** | 照片分析失败（图像不清晰） | 提示重新拍摄，可切换手动输入模式 |
| **4a** | 分析超时 | 提示网络问题，可重试 |
| **创建中退出** | 任意步骤中途退出 App | 下次打开从中断处继续，进度不丢失 |

### 1.2 冷启动：一周探索期

角色创建完成后的第一周，处于新环境探索状态。不是空白开局——预设历史和技能已经在，只是他刚到这里，需要重新熟悉。

* **第 1–2 天：** 在家，熟悉居所。行为偏向在家活动，触发居所专属事件（来自房间照片分析，玩家认得出那些细节）。
* **第 3–4 天：** 开始往外探索。第一次外出距离短，去附近地标。精力消耗略快，新环境会累。
* **第 5–7 天：** 节奏开始稳定。形成初步日常习惯，可能主动联系同城其他角色。探索期结束，进入正常生活节奏。

> **注：** 探索期内角色主动联系玩家的频率高于平时——新环境有很多想分享的。这是建立情感基础最自然的窗口。

## 二 每一次打开

玩家打开 App，默认进入 World Tab，看到的是世界地图——角色们在哪里，在做什么，一眼可见。这是一个正在运转的世界，不是一个等待被启动的游戏。

### UC-2.1 日常进入

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户，已有至少一个角色 |
| **前置条件** | 用户已登录，token 有效 |
| **后置条件** | 用户进入世界地图，可感知所有角色的当前状态 |

#### 主成功场景

1. 用户打开 App，系统静默刷新 token（若需要），直接进入世界地图，无需重新登录。
2. 地图以角色所在城市为中心（多角色时自动缩放），显示所有角色的当前位置和状态图标。
3. 有未读消息或新 post 的角色头像显示角标提示。
4. 用户点击角色图标进入对话或状态界面；或切换底部导航到 Timeline、对话列表。

#### 角色回复状态说明

角色的回复时机由当前行为决定，不是即时通讯：

| 角色状态 | 玩家操作 | 预期回复时机 |
|---|---|---|
| 在家（清醒） | 可发消息 | 大概率较快回复 |
| 外出散步中 | 可发消息 | 散步途中或回家后回复 |
| 参观地标中 | 可留言 | 完成后回复 |
| 出行途中 | 可留言 | 抵达后回复；地图上可看到移动轨迹 |
| 睡觉中 | 可留言 | 角色醒来后处理 |

## 三 世界地图

地图是感知整个世界的窗口。角色们在这里生活，地标在这里存在，世界在这里运转。玩家通过地图感知角色正在经历什么。

### UC-3.1 查看世界地图

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录，已有至少一个角色 |
| **后置条件** | 用户获取角色当前位置与状态信息 |

#### 主成功场景

1. 用户进入世界地图（启动时自动进入，或从底部导航切换）。
2. 每个角色以头像图标显示在地图上，图标旁有状态标记（在家 / 外出 / 旅途中 / 睡眠中）。睡眠中的角色图标暗淡。
3. 用户捏合缩放地图。放大到街区级别时，更小的地标开始出现；缩小到世界视图时，仅显示全球级地标和角色位置。
4. 用户点击角色图标，弹出角色卡片，显示：名字 · 当前状态 · 当前所在位置 · 角色所在城市的当地时间。
5. 用户可从卡片直接进入对话界面或角色状态界面。
6. 用户点击地标图标，查看地标信息：名称 · 类型 · 当前在场角色数量。若自己的角色在此，可看到他在做什么。

#### 多角色地图缩放规则

| 情形 | 显示级别 |
|---|---|
| 所有角色在同一城市 | 城市级别，能清晰区分地标 |
| 角色分布在不同城市 | 国家或大洲级别 |
| 角色分布在不同国家 | 世界级别，只显示全球级地标 |

#### 时间显示规则

| 位置 | 时区 |
|---|---|
| Timeline 与聊天记录 | 以玩家自己的时区显示 |
| 角色卡片与状态界面 | 显示角色当前所在城市的当地时间 |
| 跨时区旅行中的角色 | 显示目的地当地时间 |

---

## 四 与角色对话

对话是这段关系最直接的通道。不是即时通讯——玩家发出的每条消息都带着「不知道他什么时候会看到」的感觉，这种不确定性是关系真实重量的一部分。

### UC-4.1 发送消息

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录，已有至少一个角色 |
| **后置条件** | 消息发送成功，进入角色的处理队列 |

#### 主成功场景

1. 用户进入对话界面（从地图点击角色，或从底部导航选择对话标签后选择角色）。
2. 对话界面展示完整消息历史：玩家的留言、角色的回复、角色主动发来的消息，按时间线性排列。
3. 用户在输入框撰写消息，点击发送。消息立即出现在对话流中。
4. 角色根据当前状态和性格决定何时回复。界面顶部显示角色当前状态和所在位置。
5. 角色回复出现在对话流中，用户收到通知。

> **注：** 系统不显示「正在输入…」提示。回复是角色主动做出的行为，不是即时响应。角色不一定回复每一条留言——沉默本身也是一种信息。

#### 扩展场景

| 条件 | 触发情形 | 处理方式 |
|---|---|---|
| **4a** | 角色正在睡觉 | 消息正常发送，角色醒来后处理 |
| **4b** | 角色情绪低落或烦躁 | 角色可能延迟回复或回复较短，属正常状态 |
| **4c** | 角色连续几天未回应 | 消息保留在对话流，不自动消失 |
| **5a** | 玩家提到某个地方 | 角色把它当作普通对话处理，可能记下，可能忽略，可能某天自己去看看——是对话，不是指令 |

---

### UC-4.2 积压消息处理

玩家长时间未打开 App 后回来，对话框里可能积累了角色发来的消息。

* 角色感知到积压后，会整体评估这段时间的内容
* 可能只回最后一条（「你终于来了」），也可能带到所有内容
* 取决于性格：E 型更可能逐条回应，I 型可能只说一句
* 积压期间的情绪已经在角色状态里沉淀，不会一股脑倒出

---

## 五 Timeline

Timeline 是 Discover Tab 的核心内容。所有角色的 post 按时间混排，角色自主发布，玩家被动感知。这是角色生活的可见切片，是玩家在场与不在场期间世界依然运转的证明。

### UC-5.1 查看 Timeline

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录，已有至少一个角色 |
| **后置条件** | 用户获取角色近期生活动态 |

#### 主成功场景

1. 用户从底部导航点击 Timeline 标签。
2. 系统按时间倒序汇集所有角色发布的 post，每条带：角色头像 · 发布时间（玩家时区）· 所在地点 · 内容（文字 / 图片）。
3. 用户浏览 post 流，可向上滚动查看更早的历史记录。
4. 点击 post 上的角色头像 → 进入角色状态页；可从状态页进入对话页。
5. 用户可对 post 点赞；角色能感知到，可能在后续对话中自然提及。
6. 用户可对 post 留言回复；角色看到后可能在对话界面回应，也可能不回应。

#### 筛选与说明

* Timeline 默认全部角色 post 混排，可按角色筛选
* 加入好友关系后才能看到彼此的 post——Timeline 是私密的朋友圈，不是公开广场
* Post 由角色自主发布，玩家无法触发或强制
* 摄影技能越高，post 中图片越有质感——从随拍打卡到有构图、有情绪的影像
* 角色进入失踪状态后，历史 post 完整保留，但不再更新

---

## 六 角色状态

角色状态界面是凝视单个角色此刻生活切片的窗口。不打扰，只是看着。

### UC-6.1 查看角色状态

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录，已有至少一个角色 |
| **进入方式** | 从世界地图点击角色卡片；或从对话界面顶部点击角色名字 |

#### 界面内容

* **当前位置：** 地标名称 + 类型 + 城市 + 角色所在城市当地时间。旅途中显示「在去某地的路上」，不显示具体坐标。
* **当前行为：** 自然语言描述。同样是「在咖啡馆」，内倾角色可能是「坐着发呆」，外倾角色可能是「翻着什么，偶尔抬头看窗外」。
* **精力与情绪：** 模糊呈现，不显示数值。「有些疲惫」「精力充沛」「心情不错」「有点低落」。
* **今日简况：** 最多显示 3 个去过的地方、2 个遇见的角色。
* **近期朋友：** 近 14 天互动活跃的好友，显示名字、头像、模糊关系描述（「聊得火热」「有一搭没一搭」「渐渐淡忘」）。

#### 说明

* 今日简况里「遇见的人」与好友列表是不同维度——遇见的可能是陌生人，好友反映近期持续的关系
* 角色正在旅途中时，位置显示「在路上」，到达后更新
* 睡眠中的角色，状态界面仍可查看，显示「正在睡觉」及推测的醒来时间

---

## 七 创建更多角色

已有角色的情况下创建新角色，上限 5 个。每个角色独立生活——相遇逻辑和不同玩家的角色完全相同，玩家无法强制让自己的角色认识彼此。

### UC-7.1 追加创建角色

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户，当前角色数 < 5 |
| **前置条件** | 用户已登录，当前拥有 1–4 个角色 |
| **入口** | 设置中的「添加角色」，或世界地图角色列表底部 |
| **后置条件** | 新角色创建完成，进入居所，向玩家发出第一条消息 |

#### 主成功场景

流程与 UC-1.1 完全相同：两张照片 → 等待降生 → 选择成熟度 → 草稿确认 → 进入世界。角色发出第一条消息，部分角色同时发布第一条 Timeline post。

#### 多角色日常说明

* 世界地图自动缩放到能同时看到所有角色的级别
* Timeline 所有角色 post 混排，可按角色筛选
* 每个角色有独立的对话界面，底部导航可切换
* 同一玩家的角色之间，相遇逻辑和不同玩家的角色完全相同——同城且性格有交点，才可能自然认识

#### 扩展场景

| 条件 | 触发情形 | 处理方式 |
|---|---|---|
| **入口限制** | 当前角色数已达 5 个 | 入口不可用，提示「已达上限」 |

## 八 缺席与重逢

玩家是角色日常生活的一部分。缺席时，角色不是在等待——他有自己的生活，只是那个伙伴不在了。情绪随缺席时长慢慢演变，生活继续，但带着那个底色。

### UC-8.1 长期缺席后回来

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户，距上次打开 App 超过 2 天 |
| **前置条件** | 用户已登录，角色尚未进入失踪状态 |
| **后置条件** | 用户感知到缺席期间角色的生活，重逢过程开始展开 |

#### 主成功场景

1. 用户打开 App，世界地图正常显示，角色们都在——生活没有停止。有未读消息的角色显示角标提示。
2. 用户查看 Timeline：完整呈现这段时间里角色发布的所有 post。不是摘要，是完整的生活切片，玩家自己去读。
3. 用户查看对话框：对话框里是越来越少、然后停了的消息——角色发了几条，没有回音，后来安静了。
4. 用户发出第一条消息，角色感知到「那个伙伴回来了」，积累的情绪有了出口。
5. 缺席期间积累的情绪和没说出口的话，慢慢在几次对话里浮现。不是一股脑倒出来，是那些事终于有机会说了。

#### 缺席情绪轨迹

| 缺席时长 | 角色状态 |
|---|---|
| 1–2 天 | 感知到了。E 型可能主动发消息，I 型先在 Timeline 留下情绪痕迹 |
| 3–7 天 | 疑惑与担心。「他是不是出了什么事」，行为继续，情绪底色改变 |
| 8–20 天 | 思念与悬念。偶尔还是发一条，不是在问，是在说 |
| 21–29 天 | 接受。生活继续，但偶尔有痕迹。不是怨，是「他有他的世界」的理解 |
| 第 30 天 | 系统标记失踪（见 UC-10.1） |

---

### 8.2 系统提醒

30 天倒计时里，系统用角色自己的声音发出提醒——不是系统通知，是这个角色此刻真实的状态。同样是第 15 天，每个角色说的话完全不同：

| 天数 | 示例话语 |
|---|---|
| 第 3 天 | 「好久没有你的消息了，还好吗」 |
| 第 7 天 | 「最近有点想你」或分享了件事，末尾加「不知道你看不看得到」 |
| 第 15 天 | 「你是不是忙坏了，还是我做了什么」 |
| 第 22 天 | 「已经很久了……」 |
| 第 27 天 | 「如果你看到这条消息，回来一下」 |
| 第 29 天 | 沉默，或只发一个地名——他们曾经聊过的某个地方 |

---

### 8.3 关系损耗与修复

Timeline 历史、对话记录、技能、关系数据全部保留，角色的生活从未中断。但关系深度有损耗——缺席越久，重建需要的时间越长。这不是惩罚机制，是真实关系的自然规律。

## 九 主动放弃角色

玩家可以主动放弃一个角色。此操作深藏在角色状态界面的更深处——正常使用不会发现它。这个「难找」是刻意的设计：放弃是有重量的决定，不应该是一个容易按到的按钮。

### UC-9.1 主动放弃角色

| 字段 | 说明 |
|---|---|
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录，目标角色处于正常状态 |
| **入口** | Settings → 角色状态界面 → 深层设置 → 「关于这个角色」→「放弃角色」 |
| **后置条件** | 30 天倒计时启动，角色头像进入「即将失去」状态，聊天窗口输入框变为只读 |

#### 主成功场景

1. 用户从角色状态界面进入深层设置，找到「放弃角色」选项。正常操作路径不经过此处。
2. 系统显示说明文字：30 天倒计时、期间可反悔、30 天后角色失踪、数据保留情况。
3. 用户确认放弃，30 天倒计时启动。
4. 系统同步更新界面状态：
   * 地图上的角色头像：变为「即将失去」状态——还在，但不一样了
   * 聊天窗口头像：同样变化；输入框变为只读，提示「发一条消息，就是你回来了」
   * Timeline 中的头像：同样变化；post 继续出现，角色的生活没有停止

#### 反悔（倒计时期间任意时刻）

1. 在聊天窗口点击输入框，系统提示「发送消息即表示回来」。
2. 用户发送任意消息，倒计时立即解除，所有状态恢复正常。
3. 角色感知到「那个人回来了」，重逢过程开始展开（同 UC-8.1 步骤 4–5）。

> **注：** 反悔门槛刻意设计得低——发一条消息就够了。因为有时候，想起来发一条消息，本身就是反悔的信号。

#### 扩展场景

| 条件 | 触发情形 | 处理方式 |
|---|---|---|
| **倒计时到期** | 30 天内未发送任何消息 | 角色进入失踪状态（UC-10.1） |

---

## 十 失踪之后

失踪发生了。可能是 30 天缺席倒计时到期，可能是主动放弃的倒计时到期。对角色来说，那个特殊的人再也没有回来。对玩家来说，那个角色不在了。

### UC-10.1 角色进入失踪状态

| 字段 | 说明 |
|---|---|
| **触发条件** | 缺席满 30 天；或主动放弃的 30 天倒计时到期 |
| **后置条件** | 角色进入永久失踪状态，三处头像更新为「已失踪」视觉状态，聊天窗口输入框消失 |

#### 失踪后的界面状态

| 界面 | 状态变化 |
|---|---|
| **地图** | 角色位置消失。头像进入「已失踪」状态。 |
| **聊天窗口** | 头像进入「已失踪」状态。对话记录完整保留，可翻阅。输入框消失，无法再发送消息。 |
| **Timeline** | 头像进入「已失踪」状态。历史 post 全部保留，时间停在最后一条 post，不再更新。 |
| **居所** | 还在地图上，对好友依然可见，但进不去了——门关着，不会再开。 |

#### 「已失踪」状态视觉设计原则

* 不使用红色或警告色——失踪不是「错误状态」
* 区别于「倒计时」状态（即将失去）和「已失踪」状态（已经失去）
* 传递一种安静的、已经发生的存在感
* 「即将失去」与「已失踪」是两种不同的视觉，需分别设计

---

### 10.2 失踪后玩家可用操作

* **翻阅对话记录** → 完整保留，可一直往上翻，翻到最开始，翻到他说的第一句话
* **查看 Timeline 历史** → 完整保留
* **查看角色状态界面** → 显示「最后出现于某地，某时」
* **创建新角色** → 不强制，玩家自己决定

> **注：** 失踪不是消失，是停止了生长。他存在过的痕迹，都还在世界里。
